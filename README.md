# GoBid

GoBid is a real-time auction backend built in Go.  
It exposes a REST API for user and product workflows, uses WebSocket rooms for live bidding, and persists data in PostgreSQL.

## Overview

This application implements:

- User registration and login
- Session-based authentication (server-side sessions)
- Product creation with auction end time
- Real-time auction rooms using WebSockets
- Bid validation against base price and current highest bid

The application uses a layered design:

- HTTP/API layer (`internal/api`)
- Business service layer (`internal/services`)
- SQL access layer generated by `sqlc` (`internal/store/pgstore`)

## Architecture

Request flow:

1. `cmd/api/main.go` initializes configuration, PostgreSQL pool, session store, router, services, and WebSocket upgrader.
2. `internal/api` handlers decode/validate JSON payloads and enforce authentication.
3. `internal/services` implements business rules (user auth, product creation, bid rules, auction room lifecycle).
4. `internal/store/pgstore` executes type-safe SQL queries generated from `.sql` files.
5. PostgreSQL stores users, sessions, products, and bids.

Real-time flow:

1. Authenticated users connect to `GET /api/v1/products/ws/subscribe/{product_id}`.
2. The server creates/reuses an in-memory room for that product.
3. Clients send JSON messages (e.g., place a bid).
4. Room logic validates and persists bids, then broadcasts auction events to connected clients.

## Tech Stack

- Language: Go (`go 1.25.0`)
- HTTP routing/middleware: `chi` (`github.com/go-chi/chi/v5`)
- Database: PostgreSQL
- DB driver/pool: `pgx/v5` + `pgxpool`
- Typed SQL generation: `sqlc`
- Session management: `scs/v2` with PostgreSQL session backend via `pgxstore`
- Real-time transport: Gorilla WebSocket
- Security/auth helpers: `bcrypt` (`golang.org/x/crypto`) and server-side sessions
- Environment loading: `godotenv`
- Database migrations: `tern` (executed by `cmd/terndotenv`)
- Containerized local DB: Docker Compose

From `go.mod`:

- `github.com/alexedwards/scs/v2`
  - Session manager used to keep authenticated state across requests.
- `github.com/alexedwards/scs/pgxstore`
  - PostgreSQL-backed storage implementation for SCS sessions (`sessions` table).
- `github.com/go-chi/chi/v5`
  - Router for versioned API paths and middleware composition.
- `github.com/google/uuid`
  - UUID type for domain identifiers (users, products, bids).
- `github.com/gorilla/websocket`
  - WebSocket upgrade and bidirectional event handling for live auctions.
- `github.com/gorilla/csrf`
  - Included dependency; CSRF middleware/token endpoint are currently scaffolded/commented.
- `github.com/jackc/pgx/v5`
  - PostgreSQL driver and connection pool (`pgxpool`).
- `github.com/joho/godotenv`
  - Loads `.env` values into process environment in development.
- `golang.org/x/crypto`
  - `bcrypt` hashing and password verification.

Note: both `github.com/go-chi/chi` (v1) and `github.com/go-chi/chi/v5` are listed in `go.mod`.

## Database

The system uses PostgreSQL with four main tables:

- `users`
  - Stores account identity and hashed password.
  - `user_name` and `email` are unique.
- `sessions`
  - Backing store for SCS session tokens/data/expiry.
  - Includes an index on `expiry`.
- `products`
  - Auctioned items; references `users(id)` as seller.
  - Includes auction end timestamp and sold flag.
- `bids`
  - Individual bids with amount and timestamp.
  - References both `products(id)` and `users(id)`.

Schema source files:

- `internal/store/pgstore/migrations/001_create_users_table.sql`
- `internal/store/pgstore/migrations/002_create_sessions_table.sql`
- `internal/store/pgstore/migrations/003_create_products_table.sql`
- `internal/store/pgstore/migrations/004_create_bids_table.sql`

## Database Versioning (Migrations)

Database versioning is handled with **tern** migration files.

- Config file: `internal/store/pgstore/migrations/tern.conf`
- Migration format: each file contains an `up` section and a `down` section separated by:
  - `---- create above / drop below ----`

How migrations are executed in this project:

- `cmd/terndotenv/main.go` loads `.env`, resolves migration/config paths, and runs:
  - `tern migrate --migrations <path> --config <path>`

This keeps schema changes versioned, reproducible, and reversible.

## How `pgx`, `pgxpool`, and `pgxstore` Fit Together

- `pgx` (`github.com/jackc/pgx/v5`)
  - Low-level PostgreSQL driver used by application queries.
  - Handles query execution, row scanning, errors, and transaction primitives.

- `pgxpool`
  - Connection pool from `pgx`.
  - Reuses database connections efficiently and is shared by all services/queries.

- `pgxstore` (`github.com/alexedwards/scs/pgxstore`)
  - Session store adapter for SCS using the same PostgreSQL pool.
  - Reads/writes session tokens and payloads in the `sessions` table.

In `cmd/api/main.go`:

1. A `pgxpool.Pool` is created and pinged.
2. `sqlc` queries (`pgstore.New(pool)`) and business services use that pool.
3. `scs` is configured with `pgxstore.New(pool)` for persistent session storage.

## API Endpoints

Base prefix: `/api/v1`

User endpoints:

- `POST /users/signup`
  - Creates a new user.
- `POST /users/login`
  - Authenticates user and stores `AuthenticatedUserId` in session.
- `POST /users/logout` (authenticated)
  - Renews token and clears session auth key.

Product/Auction endpoints:

- `POST /products/` (authenticated)
  - Creates a product and starts auction room lifecycle.
- `GET /products/ws/subscribe/{product_id}` (authenticated)
  - Upgrades to WebSocket and subscribes user to product auction room.

## WebSocket Auction Protocol

Message model (`internal/services/auction_services.go`):

- `kind` (numeric enum)
- `message` (optional text)
- `amount` (bid value when relevant)
- `user_id` (server-side identity context)

Important message kinds:

- `PlaceBid` (client request)
- `SuccessfullyPlacedBid` (ack to bidder)
- `FailedToPlaceBid` (validation error, e.g., bid too low)
- `NewBidPlaced` (broadcast to other room clients)
- `InvalidJson` (input parse failure)
- `AuctionFinished` (room deadline reached)

Room behavior:

- One active room per product in `AuctionLobby.Rooms`.
- Room context uses a deadline equal to `auction_end`.
- On expiration, room broadcasts auction completion and closes client channels.

## Validation and Error Handling

- Request-level validation is implemented in `internal/usecase/*` DTOs via `Valid(context.Context)`.
- `internal/validator` provides reusable checks (not blank, length, email regex).
- `jsonutils.DecodeValidJson` decodes and validates payloads in one step.
- Service layer maps technical errors to domain errors:
  - duplicate user -> `ErrDuplicatedEmailOrUsername`
  - invalid credentials -> `ErrInvalidCredentials`
  - product not found -> `ErrProductNotFound`
  - low bid -> `ErrBidIsTooLow`

## Local Development Setup

### Prerequisites

- Go 1.25+
- Docker + Docker Compose
- `tern` CLI installed and available in `PATH`

### 1) Configure environment

Create `.env` with:

```env
GOBID_DATABASE_PORT=5580
GOBID_DATABASE_NAME=gobid
GOBID_DATABASE_USER=postgres
GOBID_DATABASE_PASSWORD=your_password
GOBID_DATABASE_HOST=localhost
GOBID_CSRF_KEY=replace_with_32_plus_random_bytes
```

### 2) Start PostgreSQL

```bash
docker compose up -d
```

### 3) Run migrations

```bash
go run ./cmd/terndotenv
```

### 4) Run API server

```bash
go run ./cmd/api
```

Server address:

- `http://localhost:8080`

## Common Commands

Run tests:

```bash
go test ./...
```

Regenerate SQL code after editing SQL/migrations:

```bash
sqlc generate -f ./internal/store/pgstore/sqlc.yml
```

Inspect users table inside Dockerized Postgres:

```bash
docker compose exec db psql -U postgres -d gobid -c "\d users"
```

Optional live reload example (as used in project notes):

```bash
air --build.cmd "go build -o ./bin/api.exe ./cmd/api" --build.bin ".\\bin\\api.exe"
```

## Current Limitations and Improvement Opportunities

- CSRF protection is present but currently commented out in route setup.
- Some error paths in bid service return empty responses instead of explicit errors.
- No automated test suite yet (only compilation checks).
- API documentation can be further improved with OpenAPI/Swagger definitions.
